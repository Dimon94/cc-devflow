# CC-DevFlow v4.3.0 Migration Guide

## Overview

This guide is for teams moving from the old slash-command and harness-heavy mental model to the current skills-first distribution model.

The key change is simple:

- visible workflow surface: skills
- distribution surface: `cc-devflow init` and `cc-devflow adapt`
- harness: internal implementation detail

---

## What Changed

### 1. Workflow Entry Changed

Old mental model:

```text
/flow:init -> /flow:spec -> /flow:dev -> /flow:verify -> /flow:release
```

Current mental model:

```text
roadmap

req-plan -> req-do -> req-check -> req-act
```

The workflow is now represented by explicit skills rather than a slash-command namespace.

### 2. Distribution Entry Changed

Whole-pack install and platform adaptation now belong to the packaged CLI:

```bash
npx cc-devflow init --dir /path/to/project
npx cc-devflow adapt --cwd /path/to/project --platform codex
```

### 3. `skills.sh` Scope Narrowed

`skills.sh` remains supported, but only for single-skill distribution:

```bash
npx skills add https://github.com/Dimon94/cc-devflow --skill roadmap
```

Do not use `skills.sh` as the whole-pack lifecycle anymore.

### 4. `.claude/commands/` Is Optional

The compiler still accepts legacy prompt inputs if a project has `.claude/commands/`, but the repository no longer requires that directory to exist.

Skills-first repositories are valid.

### 5. Harness CLI Is Gone

`harness:*` is no longer the public CLI surface.

Harness code may still exist under `lib/harness/`, but it should be treated as an internal runtime library.

---

## Old To New Mapping

| Old Concept | New Concept |
|-------------|-------------|
| `/flow:*` | visible skills |
| `harness:*` CLI | internal runtime only |
| clone `.claude` manually | `cc-devflow init` |
| per-platform handwritten setup | `cc-devflow adapt` |
| skills pack via old CLI lifecycle | whole pack via `cc-devflow`, single skill via `skills.sh` |

---

## Migration Checklist

1. Replace old workflow references in docs with the five visible skills.
2. Replace old pack-install steps with `cc-devflow init`.
3. Replace old platform setup steps with `cc-devflow adapt`.
4. Keep `skills.sh` only for single-skill install or refresh.
5. Treat `.claude/commands/` as optional compatibility input, not required structure.
6. Stop documenting `harness:*` as if users should run it directly.

---

## Verification

After migration, these commands should succeed:

```bash
npx cc-devflow --help
npx cc-devflow init --dir /tmp/cc-devflow-demo
npx cc-devflow adapt --cwd /tmp/cc-devflow-demo --platform codex
```

And the installed project should contain:

```text
.claude/skills/roadmap/SKILL.md
.claude/skills/req-plan/SKILL.md
.claude/skills/req-do/SKILL.md
.claude/skills/req-check/SKILL.md
.claude/skills/req-act/SKILL.md
```
