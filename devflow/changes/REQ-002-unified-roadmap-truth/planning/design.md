## Options Considered

### Option A: Minimal viable migration

- Shape: rename `roadmap-tracking.json` to `roadmap.json`, keep the current v2 item model, render `ROADMAP.md`, and keep `BACKLOG.md`.
- Reuses: current schema, markdown parser, renderer, locator, sync scripts.
- Pros: fastest implementation and least test churn.
- Cons: preserves the old sidecar-shaped data model and keeps strategic context partly in Markdown.
- Risks: future agents still need to ask which file owns roadmap strategy.

### Option B: Unified roadmap state

- Shape: introduce v3 `roadmap.json` with `meta`, `context`, `evidence`, `route`, `stages`, `items`, `handoff`, and `architecture`; render `ROADMAP.md` and deprecated `BACKLOG.md`.
- Reuses: current item normalization, legacy markdown parser, render command, query path, and examples.
- Pros: one editable machine truth, generated views, data-driven diagrams, and a clear migration path.
- Cons: touches more files and requires careful upgrade logic.
- Risks: schema can become too broad if implementation tries to model every future roadmap idea now.

### Option C: Roadmap package directory

- Shape: move roadmap internals under `devflow/roadmap/state.json` and generated views under the same directory.
- Reuses: renderer internals but not the current path convention.
- Pros: clean grouping for a larger future subsystem.
- Cons: reopens a previously reversed path migration and increases downstream disruption.
- Risks: path churn hides the actual value of removing duplicate truth.

## Approved Direction

- Approved option: Option B, Unified roadmap state.
- Why this is the best trade-off now: it removes the duplicated truth boundary instead of just renaming it, while preserving a one-release compatibility projection for downstream users.
- Why not the other options now:
  - Option A is too shallow because it keeps the old item-only sidecar model.
  - Option C is too disruptive because `devflow/roadmap/*` was already moved back to `devflow/*`.
- What we explicitly rejected:
  - permanent first-class `BACKLOG.md`
  - hand-written Mermaid architecture diagrams in `ROADMAP.md`
  - a new canonical `devflow/roadmap/` directory
  - removing all legacy migration behavior in the first release
- Frozen decisions:
  - new machine truth path is `devflow/roadmap.json`
  - schema version is `3`
  - `BACKLOG.md` is a deprecated generated projection for one compatibility release
  - `cc-roadmap` major version becomes `5.0.0`
  - architecture diagrams are generated from flat `nodes` and `edges`
  - legacy input priority is `roadmap.json` first, then `roadmap-tracking.json`, then markdown fallback
- Deferred questions:
  - exact release where deprecated `BACKLOG.md` output is removed
  - grouped architecture subgraphs
  - downstream repo upgrade execution

## Design

### Data Flow

```text
legacy ROADMAP/BACKLOG or roadmap-tracking.json
  -> loadRoadmapState()
  -> v3 roadmap state
  -> persistRoadmapFiles()
  -> devflow/roadmap.json
  -> devflow/ROADMAP.md
  -> devflow/BACKLOG.md (deprecated projection)
```

### Target Architecture Diagram

```text
specs / changes / decisions
  -> roadmap.json
  -> ROADMAP.md
  -> Ready For cc-plan
  -> cc-plan
  -> cc-do
  -> cc-check
  -> cc-act
  -> roadmap.json
```

The actual Mermaid diagram in generated roadmap output must come from
`roadmapState.architecture.nodes` and `roadmapState.architecture.edges`.

### Roadmap State Contract

`roadmap.json` version 3 stores complete roadmap semantics:

```json
{
  "version": 3,
  "outputPolicy": { "documentLanguage": "en" },
  "meta": {
    "roadmapVersion": "roadmap.v1",
    "skillVersion": "5.0.0",
    "status": "active",
    "lastUpdated": "2026-05-01",
    "currentFocusStage": "Stage 1"
  },
  "context": {
    "planningPosture": "infrastructure",
    "evidenceMaturity": "infra-only",
    "canonicalTerms": [],
    "durableDecisionSources": []
  },
  "evidence": [],
  "route": {
    "recommended": "platform-first",
    "whyThisWinsNow": "",
    "rejectedRoutes": [],
    "firstSignal": "",
    "killSignal": ""
  },
  "stages": [],
  "items": [],
  "handoff": {
    "readyForCcPlan": [],
    "parked": [],
    "serialSpine": [],
    "parallelWaves": []
  },
  "architecture": {
    "diagramType": "flowchart",
    "nodes": [],
    "edges": []
  }
}
```

Implementation should keep the first v3 schema direct. Do not add generic DSL features before a real roadmap needs them.

### Public Interfaces

- Existing command name can remain `roadmap-tracking.js` for compatibility, but help text should introduce `roadmap.json` as the default state path.
- `render` should require `--roadmap` and `--state` or continue accepting `--tracking` as an alias during migration.
- `sync-roadmap-progress.sh` should default to `devflow/roadmap.json`.
- `locate-roadmap-item.sh` should default to `devflow/roadmap.json`, then fall back to `devflow/roadmap-tracking.json`, repo-root legacy state, and legacy markdown.
- Generated `BACKLOG.md` must include a top-screen deprecation notice.
